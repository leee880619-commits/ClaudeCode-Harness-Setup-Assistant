# Pipeline Review Gate

## 원칙
대상 프로젝트의 **생성·결정·설계 출력** 파이프라인은 마지막 스텝으로 **도메인 전문 레드팀 리뷰어**를 필수 포함. Phase 4에서 강제하고, Phase 5·6이 실현.

## 파이프라인 분류

### 리뷰 필수 (mandatory review)
다음 중 하나라도 해당하면 리뷰어 스텝 포함이 **의무**:
- **생성 (generation)**: 새로운 콘텐츠·문서·코드·다이어그램·카피를 생성
- **결정 (decision)**: 후보 중 선택, 우선순위 책정, 전략 분기
- **설계 (design)**: 아키텍처·API·스키마·워크플로우 초안 작성
- **계획 (planning)**: 구현 계획, 일정, 리소스 배분
- **리서치 (research)**: 외부 정보 수집 후 요약/해석 (출처 신뢰성 검증 필요)

### 리뷰 면제 가능 (exempt-eligible)
아래는 면제가 가능하나, `review_exempt: true + exempt_reason` 명시 필수:
- **결정론적 변환 (transformation)**: 포맷팅, 린트, 마이그레이션 스크립트 실행
- **단순 I/O**: 파일 이동·복사·삭제, git 커밋, 빌드 실행
- **검증·테스트 실행 (execution only)**: 기존 테스트 스위트 실행, 결과 리포트. 단, 테스트 **설계** 파이프라인은 "생성"이므로 리뷰 필수.
- **조회·읽기 (read-only)**: 로그 조회, DB 조회, API 조회 후 원문 전달

**면제 판단의 권위**: 설계자(Phase 4 에이전트)가 분류하되, 메타 레벨 Advisor(Dim 12)가 면제 사유를 재검증한다. 애매하면 면제하지 않는다.

## 리뷰어 요구 사항

1. **도메인 특화**: 범용 Advisor 1명으로 모든 파이프라인을 커버하지 않고, 주제별 리뷰어를 분리 프로비저닝. 예: `research-pipeline` → `research-redteam` (출처 신뢰성·편향)
2. **리뷰 전용**: 리뷰어는 파일 생성·수정 권한을 갖지 않는다. 결과는 BLOCK/ASK/NOTE 구조화 리포트로만 반환.
3. **재귀 금지**: 리뷰어 자체의 출력은 다시 리뷰받지 않는다. "리뷰의 리뷰" 금지.
4. **단수 원칙**: 한 파이프라인에 리뷰어는 1명 (기본). 복수가 필요하면 파이프라인을 분할하거나 Escalations에 근거 기록.
5. **입력 범위**: 리뷰어는 파이프라인 최종 출력 + 사용자의 원 의도만 받는다. 파이프라인 내부 중간 산출물은 기본 비공개.

## 에스컬레이션 래더 (BLOCK 처리 프로토콜)

리뷰어가 BLOCK을 반환하면 파이프라인을 재실행한다. 재실행은 재리뷰를 유발한다. 각 재리뷰 요청은 아래 래더에 따라 처리한다.

| 회차 | BLOCK 회수 | 재리뷰 요청 처리 | 동작 |
|------|-----------|-----------------|------|
| 1 | 1회 | **메인 오케스트레이터가 판단 후 자동 승인** | BLOCK 사유·제안 반영하여 파이프라인 재실행. 사용자 중단 없음. |
| 2 | 2회 | **사용자에게 보고 후 의사결정 요청** (AskUserQuestion) | (A) 재실행 (B) BLOCK 수용 후 진행 (C) 수동 편집 — 사용자 답변에 따라 분기 |
| 3 | 3회 | **작업 전면 중단 + 사용자 3선택** (AskUserQuestion) | (1) 무시하고 진행 (2) 수동 개입 (사용자가 산출물 직접 편집 후 "편집 완료" 응답) (3) 해당 파이프라인 스킵 — 하류 파이프라인은 제한된 맥락으로 진행 |

### 래더 세부 규약

- **카운팅 기준**: 동일 파이프라인 1회 호출 내에서 누적. 파이프라인이 완료되어 다음 호출로 넘어가면 카운터 리셋.
- **1회차 자동 승인 조건**: 오케스트레이터가 "BLOCK 사유가 파이프라인 스코프 내에서 해결 가능하며, 재실행 시 성공 가능성이 합리적"이라고 판단한 경우에만 자동 승인. 판단이 서지 않으면 즉시 사용자 에스컬레이션(2회차 동작 선취).
- **3회차 중단 시 기록**: 파이프라인의 상태 저장소(예: 대상 프로젝트 `docs/` 또는 파이프라인 상태 파일)에 `review_escalation: halted_at_round_3, unresolved_blocks: [...]` 형태로 기록. 재개 시 이 상태를 우선 확인.
- **카운터 저장 위치**: 런타임 카운터는 파이프라인 실행 컨텍스트(메인 세션 메모리 또는 상태 파일)에 보관. Phase 4 설계 산출물은 "이 래더를 따른다"는 규약만 명시.
- **래더 우회 금지**: 설계자는 리뷰어에 자체 재시도 루프를 심어 래더를 우회하지 않는다. 재시도는 오직 이 래더를 통해서만 발생.

## Phase별 책임

| Phase | 이 규칙 관련 책임 |
|-------|-----------------|
| 4 `pipeline-design` | 각 파이프라인을 "리뷰 필수 / 면제" 분류, 리뷰어 스텝을 파이프라인 말단에 배치, 면제 시 `review_exempt: true + exempt_reason` 기록. 각 리뷰어의 도메인 스코프와 BLOCK/ASK/NOTE 기준 초안 기록. 에스컬레이션 래더는 이 문서를 참조(복붙 금지). |
| 5 `agent-team` | Phase 4가 지정한 도메인 리뷰어마다 에이전트 파일(`.claude/agents/{domain}-redteam.md`) 프로비저닝. 리뷰 전용이므로 `allowed_dirs` 비움 또는 read-only 경로만. |
| 6 `skill-forge` | 리뷰어 에이전트의 SKILL.md에 도메인 Dimension 체크리스트 주입 (예: research-redteam의 "출처 최신성·편향·상반 주장 누락"). |
| 9 `final-validation` | 생성·결정·설계 파이프라인 중 리뷰 스텝 누락 또는 `review_exempt` 사유 누락을 BLOCK으로 감지. 에스컬레이션 래더 문서화 유무 확인. |

## 메타 레벨 Advisor 연계
이 플러그인의 Red-team Advisor는 Phase 4 산출물 리뷰 시 **Dim 12 (파이프라인 리뷰 게이트 준수)** 를 검사한다. 구체 항목은 `red-team-advisor.md` 의 Dim 12 정의 참조.

## Guardrails
- 리뷰어 역할 정의에 "방어적 검증, 반례 탐색" 명시 (모든 파이프라인을 NOTE로 통과시키면 게이트 형해화).
- 비용이 과하면 파이프라인 분류를 재검토 — 리뷰어 스텝 제거가 아님.
- 단순 프로젝트도 생성·결정 파이프라인이면 리뷰어 필수. 복잡도 게이트로 스킵 금지.
