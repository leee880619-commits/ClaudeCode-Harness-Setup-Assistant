# 코드 구현 워크플로우 규칙

## 절대 원칙: 문제는 정면 돌파한다

**문제를 우회하거나 임시방편으로 회피하는 것을 금지한다.**
- 오류가 발생하면 증상이 아니라 **근본 원인**을 찾아 해결한다
- "일단 돌아가게만" 하는 패치, 예외 처리로 덮기, 문제가 되는 코드를 우회하는 구조 변경은 모두 금지한다
- 근본 원인을 파악하기 전에 해결책을 제시하지 않는다
- 근본 원인이 불명확하면 추가 리서치를 수행하거나 오케스트레이터를 통해 사용자에게 질문한다

---

## 실행 방식: Agent Teams

**이 워크플로우의 모든 단계는 Agent Teams를 스폰하여 수행한다.**
- 각 단계(리서치, 계획, 레드팀 검토, 구현, QA)는 독립된 전문 에이전트를 스폰하여 실행한다
- 오케스트레이터는 단계 간 산출물(research.md, plan.md 등)을 다음 에이전트에게 전달하며 흐름을 관리한다
- 에이전트 간 암묵적 정보 전달은 금지한다. 모든 컨텍스트는 파일 또는 명시적 지시로 전달한다
- 사용자와의 질문/승인 인터페이스는 오케스트레이터가 단독으로 담당한다. 개별 에이전트가 사용자에게 직접 질문하지 않는다

---

## 핵심 원칙

- **무지한 변경** 방지: 코드 리서치로 시스템을 충분히 이해한 뒤 움직인다
- **잘못된 변경** 방지: plan.md로 설계를 먼저 확정한 뒤 구현한다
- **의도·도메인 지식 주입**: 암묵적 가정 없이 모든 결정 사항을 사용자에게 질문하여 확인한다
- **무슨 코드를 쓸지**는 항상 사용자가 결정한다. 에이전트는 결정하지 않는다.
- 모든 작업은 **하나의 긴 세션**에서 끊기지 않고 실행한다
- **불명확한 상황이 발생하면 즉시 멈추고 오케스트레이터를 통해 사용자에게 질문한다. 임의로 판단하고 진행하지 않는다.**

---

## 전체 흐름 (6단계)

### STEP 0 — 폴더 구조 세팅
- `docs/` 폴더 안에 사용자 요청을 요약한 이름(kebab-case)으로 하위 폴더를 생성한다
- 이후 모든 단계의 산출물(research.md, plan.md 등)은 **해당 폴더 안에만** 생성한다
- 목적: 작업별 히스토리를 독립적으로 관리한다

### Resume Detection
- 기존에 같은 이름의 작업 폴더가 있으면 AskUserQuestion으로 "이전 작업 재개 / 새로 시작" 확인

---

### STEP 1 — 코드 리서치 에이전트 (`research.md`)
- **[스폰: researcher-agent]**
- 관련 폴더·파일·기능을 **깊이, 매우 상세히, 세부 사항까지** 분석한다
- 결과는 채팅이 아니라 반드시 `research.md` 파일로 작성한다
- 이유: 겉핥기 이해는 레이어 무시·중복 API·ORM 무시 같은 "조용히 시스템을 망가뜨리는 변경"을 낳는다

#### STEP 1-1 — 웹 리서치 에이전트 (`web_research.md`) ← 조건부 실행
아래 두 조건 중 하나라도 해당하면 웹 리서치 에이전트를 추가로 스폰한다:
- 복잡한 기능 수정 또는 신규 피처 추가 요청인 경우
- 코드 리서치 결과, 수정 계획을 세우는 데 확신이 100% 부족한 경우

- **[스폰: web-researcher-agent]** — 결과는 `web_research.md` 파일로 작성한다

---

### STEP 2 — 플래너 에이전트 (`plan.md`)
- **[스폰: planner-agent]** — research.md (및 web_research.md)를 컨텍스트로 전달하여 스폰한다
- 리서치를 검토한 뒤 상세 구현 계획 초안을 `plan.md` 파일로 작성한다
- **반드시 포함할 항목:**
  - 기능별 구현 방법 (코드 스니펫 포함)
  - 수정될 파일 경로 전체 목록
  - 고려사항과 트레이드오프
- 계획 작성 전 반드시 소스 파일을 직접 읽고, **실제 코드베이스를 기반으로** 작성한다
- **절대 이 단계에서 코드를 구현하지 않는다**

---

### STEP 3 — 질문 사이클 (plan.md 반복 정제)

plan.md 초안 작성 후, 오케스트레이터가 아래 절차를 반복한다.

#### 3-1. 질문 도출 및 질문

**[스폰: question-drafter-agent]** — plan.md와 research.md를 컨텍스트로 전달하여 질문 초안을 도출.

다음 유형에 해당하는 모든 항목을 빠짐없이 추출하여 `docs/{task-name}/questions-draft.md`로 저장:
- **가정 검증**: 사실이라고 가정하고 계획을 짰지만, 틀릴 경우 설계가 바뀌는 항목
  - 예: "이 API는 PATCH가 아니라 PUT을 사용하는 것으로 가정했습니다. 맞나요?"
- **의사결정 필요**: 구현 방향이 둘 이상이고 사용자만 선택할 수 있는 항목
  - 예: "캐싱을 적용할지 여부는 비즈니스 요구사항에 달려 있습니다. 적용할까요?"
- **도메인 규칙 확인**: 코드베이스에서 확인되지 않은 비즈니스 규칙이나 정책
  - 예: "리스트가 공개일 때 하위 아이템도 자동 공개되는 규칙이 있나요?"
- **범위 경계**: 이번 작업에 포함해야 하는지 불명확한 항목
  - 예: "관련된 알림 로직도 함께 수정해야 하나요, 아니면 이번 범위에서 제외하나요?"

오케스트레이터는 이 초안을 AskUserQuestion으로 사용자에게 **한 번에** 제시한다. 각 질문마다 **현재 가정하고 있는 기본값**을 함께 명시한다.

#### 3-2. 답변 반영
- 사용자 답변을 받으면 **[스폰: planner-agent]** 를 재스폰하여 plan.md를 즉시 업데이트한다
- 답변으로 인해 새로운 불명확한 사항이 생기면 3-1을 다시 수행한다
- 더 이상 질문이 없고 사용자도 수정 요청이 없으면 이 사이클을 종료한다

**이 단계에서도 코드를 구현하지 않는다**

---

### STEP 4 — 레드팀 에이전트 검토 (plan.md 업데이트)
- **[스폰: redteam-agent]** — 확정된 plan.md를 컨텍스트로 전달하여 스폰한다
- 검토 항목:
  - 설계의 잠재적 결함·위험 요소
  - 누락된 엣지 케이스
  - 더 나은 대안 접근 방식
  - 복잡도·유지보수성 리스크
- 검토 결과를 plan.md에 반영하고, 검토로 인해 새로운 질문이 발생하면 STEP 3-1로 돌아가 사용자에게 질문한다
- 검토 완료 결과를 제시한 뒤 수정 또는 새로 작성해야 하는 스크립트 명을 명시한다
- **이 단계에서도 코드를 구현하지 않는다**

---

### STEP 5 — 구현 에이전트
- 사용자로부터 **명시적인 구현 승인**을 받은 후에만 구현 에이전트를 스폰한다
- **[스폰: implementer-agent]** — 최종 확정된 plan.md를 컨텍스트로 전달하여 스폰한다
- 구현 시 준수 사항:
  - plan.md의 모든 작업을 순서대로 구현한다
  - 각 작업이 완료될 때마다 plan.md에 완료 표시를 한다
  - 새로운 타입을 임의로 만들지 않는다
  - 타입 체크를 지속적으로 실행한다
  - **불명확한 상황이 발생하면 즉시 멈추고 오케스트레이터를 통해 사용자에게 질문한다**

---

### STEP 6.0 — 화이트박스 QA 에이전트 (항상 실행)
- **[스폰: qa-whitebox-agent]**
- 구현 완료된 코드를 코드 수준에서 검증한다:
  - 스크립트 기반 게이트 테스트 (프로젝트 정의 명령: `npm run check:*`, `pytest`, `cargo test` 등)
  - 정적 분석 (타입 체크, 린트)
  - 코드 리뷰: 패턴 위반, 누락된 에러 처리, 데드 코드
  - plan.md 준수 여부 검증
  - 타입 일관성 체크
- **임시 패치 여부 체크** — 아래 항목을 명시적으로 검증하고 결과를 보고한다:
  - 구현된 코드 중 `try/catch`로 오류를 삼키거나 무시한 곳이 있는가
  - 근본 원인 해결 없이 증상만 억제한 조건문(예: null 체크 남용, 하드코딩 예외값)이 있는가
  - TODO·FIXME·임시(temp/hack) 주석이 남아 있는가
  - plan.md에 없던 우회 로직이 구현 중 추가되었는가
  - 위 항목 중 하나라도 발견되면 **임시 패치로 간주하고 즉시 보고한다. 사용자 승인 없이 넘어가지 않는다**
- 발견된 문제는 오케스트레이터를 통해 즉시 보고하고, 사용자 승인 후 수정한다

---

### STEP 6.1 — 블랙박스 QA 에이전트 (조건부 실행)
아래 조건 중 하나라도 해당하면 블랙박스 QA 에이전트를 스폰한다:
- 기능 수정 또는 신규 기능 추가가 포함된 경우
- UI/UX 변경이 포함된 경우 (레이아웃, 색상, 텍스트, HUD 등)
- 사용자 대면 동작이 변경된 경우 (키 입력, 게임 플로우, 네비게이션 등)
- 네트워크/WebSocket/SSE 통신이 변경된 경우
- 사용자가 명시적으로 블랙박스 테스트를 요청한 경우

블랙박스 QA를 **실행하지 않는** 경우:
- 순수 리팩토링 (동작 변경 없는 코드 구조 개선)
- 내부 설정/문서만 변경된 경우
- 화이트박스 QA에서 FAIL 판정이 나온 경우 (화이트박스 문제를 먼저 수정)

- **[스폰: qa-blackbox-agent]**
- 실제 브라우저/런타임에서 동작을 검증한다:
  - 서버/앱 기동 및 라이브 테스트
  - 기능 동작 검증 (시나리오 기반)
  - 시각적 검증 (레이아웃, 색상, 상태 표시)
  - 사용자 상호작용 테스트 (키보드, 클릭)
  - 네트워크/SSE/WebSocket 연결 검증
  - 엣지 케이스 런타임 테스트
- 발견된 문제는 오케스트레이터를 통해 즉시 보고하고, 사용자 승인 후 수정한다

---

## 문제 발생 시: 패치보다 되돌리기

- 수정 방향이 불확실하거나 설계가 틀렸다는 판단이 들면 계속 패치하지 않는다
- **`git reset` / `git revert`로 되돌린 뒤 작업 범위를 줄여서 plan.md를 새로 작성한다**
- 이유: 틀린 설계 위에서 패치를 누적하면 사용 불가능한 코드가 된다

---

## AskUserQuestion Ownership

- 모든 사용자 질문은 오케스트레이터(메인 세션)가 담당한다
- 서브에이전트는 불확실 사항을 산출물의 **Escalations** 섹션에 기록한다
- 오케스트레이터는 Escalations + question-drafter 산출물을 종합하여 일괄 AskUserQuestion 수행
