# Code Navigation via Code Map

## 코드맵 파일

기본 경로: `docs/architecture/code-map.md`

프로젝트에 다른 위치/이름을 쓰면 이 규칙의 경로를 조정한다. **이 규칙이 프로젝트에 설치되어 있다는 것은 "코드맵 기반 탐색을 쓰기로 결정했다"는 뜻**이다. 파일이 없는 상태가 지속되는 것은 정상이 아니므로, 리서치 에이전트는 **최초 1회 Escalations에 `[ASK] code-map.md 부재 — 생성 제안` 기록**을 요청한다 (매 리서치마다 반복하지 않는다).

## 코드 리서치 시 사용

코드베이스 탐색을 시작하기 **전에** 코드맵을 먼저 읽는다.

1. `docs/architecture/code-map.md` 존재 여부 확인
2. **있으면**: 사용자 요청과 관련된 파일/함수 위치를 코드맵에서 파악한 뒤, 해당 위치만 타겟팅하여 소스 파일을 Read. 코드맵에 없는 영역은 Grep/Glob으로 확장 탐색
3. **없으면**: Grep/Glob 기반 무차별 탐색으로 진행. 동시에 이 작업 폴더(`docs/{task-name}/`) 내 이전 research에서 이미 code-map 부재 Escalation이 기록되었는지 확인
   - 기록 없으면: 이번 research.md의 Escalations에 `[ASK] code-map.md 부재 — 생성할지 확인 필요` 추가
   - 이미 기록 있으면: 중복 기록하지 않음

이 순서는 전체 코드베이스를 무차별 탐색하는 것보다 빠르고 정확하며 컨텍스트 윈도우를 절약한다.

## 구현 완료 후 코드맵 업데이트

구현 작업자가 코드 변경을 커밋한 뒤, 아래 조건에 해당하면 코드맵을 업데이트한다.

### 업데이트 필요 (구조 변경)

- 함수 추가, 삭제, 이름 변경
- 함수가 다른 파일로 이동
- 새 소스 파일 추가 또는 기존 파일 삭제
- API 엔드포인트 추가, 삭제, 경로 변경
- 파일 간 의존 관계 변경 (새 import/require, 공개 API 변경)
- 새 DOM 구조 추가 (주요 섹션/라우트/페이지)

### 업데이트 불필요

- 함수 내부 로직 수정 (버그 픽스, 값 변경)
- CSS/스타일 변경
- i18n 번역 키 추가 (기능 구조 불변 시)
- 설정값/상수 조정
- 주석 변경

### 업데이트 방법

- 변경된 부분만 코드맵에 반영한다. 전체를 다시 쓰지 않는다
- 업데이트 후 코드맵 상단의 `최종 업데이트` 날짜(또는 해당 메타 필드)를 갱신한다
- 코드맵이 아직 없고 구조 변경이 발생했다면, **신규 생성은 이 규칙의 범위가 아니다** — 구현 에이전트가 Escalations에 `[ASK] code-map.md 부재 — 생성 제안` 기록 (이전 단계에서 이미 기록되었으면 중복하지 않음). 오케스트레이터가 사용자에게 질문하여 생성 여부를 결정한다

## 왜 이 규칙이 중요한가

- 대형 코드베이스에서 코드맵은 1차 진입점. 매 리서치마다 처음부터 코드베이스를 탐색하면 시간·컨텍스트 낭비
- 구조 변경 후 코드맵이 뒤처지면 다음 리서치가 오도되므로, 변경과 업데이트를 같은 작업 단위에 묶는다
- 코드맵이 없는 프로젝트에는 아무 영향 없음 — opt-in 방식
