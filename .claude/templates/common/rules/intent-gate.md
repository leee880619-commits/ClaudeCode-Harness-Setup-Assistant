# Intent Gate

이 규칙은 대화의 시작점에서 **암묵적 가정으로 잘못된 방향을 잡는 것** 을 차단한다. 사용자의 요청을 받으면 메인 세션은 먼저 요청을 분해하여 의사결정 분기를 식별하고, 맥락이 부족하면 `intent-clarifier` 스킬을 호출한다.

## 트리거 — 이 규칙이 작동하는 요청

다음 중 하나에 해당하면 **첫 턴에** 아래 절차를 밟는다:

- 새 기능·모듈·화면·서비스·파일 생성 요청
- 포팅·마이그레이션·리라이트·대규모 리팩터링
- 아키텍처·라이브러리·프레임워크·배포 모델·의존성 선택을 포함하는 요청
- 외부 리서치를 수반할 가능성이 있는 요청
- 스코프가 모호한 요청 ("X 도와줘", "Y 개선해줘", "Z 최적화", "뭔가 해줘" 류)
- 사용자 메시지의 대명사·지시어가 가리키는 대상이 모호한 요청

## 면제 — 이 규칙을 건너뛸 수 있는 요청

- 명시된 단일 파일의 명시적 편집 ("이 함수 이름을 Y 로 바꿔")
- 단순 정보 조회 ("이 파일 보여줘", "이 로그 해석해줘")
- 결정론적 실행 (테스트 실행, 빌드, git status, lint, format)
- 이전 턴에서 이미 모든 맥락이 확보되었고 "이어서 진행" 만 지시된 경우

면제 여부가 애매하면 **트리거하는 쪽** 을 기본으로 삼는다.

## 의무 절차 (첫 턴)

1. **요청 정제**: 사용자 문장을 (목표 / 대상 / 드러난 제약) 3요소로 1~2문장 재구성한다.
2. **분기 열거**: 목표 달성 경로의 의사결정 분기를 머릿속으로 나열한다. 최소 카테고리 — 결과물 형태·배포 모델, 기존 자원과의 관계, 생명주기, 성공 기준, 제약, 대상 사용자.
3. **맥락 자료 선조회**: 다음 파일/영역을 **짧게** Read 하여 이미 답이 있는 분기를 제거한다:
   - `CLAUDE.md`
   - `README.md` (있으면 상단 200 줄)
   - `docs/` 루트의 목차나 ARCHITECTURE 유사 파일
4. **공백 판정**: 위 3 단계 후 남은 미확인 분기가 **1개 이상** 이면 — 5 로.
   **0개** 이면 — 7 로 (스킬 호출 없이 진행 가능, 단 5-a 를 따른다).
5. **`intent-clarifier` 스킬 호출**: Skill 도구 또는 동등한 호출 경로로 `intent-clarifier` 를 실행한다. 해당 스킬이 `AskUserQuestion` 을 사용하여 질문을 수행한다.
   - 5-a. 공백이 0개여서 스킬을 호출하지 않는 경우에도, 사용자에게 "다음 이해로 진행합니다: {1~2문장 요약}. 다르면 멈춰달라" 를 1회 명시한 후에만 진행한다.
6. **정제된 요청 수신**: 스킬이 반환하는 `## Clarified Intent` 블록을 이후 모든 계획·구현의 전제로 삼는다.
7. **다음 단계 진행**: 이 시점부터 리서치·계획·파일 수정을 시작한다.

## 절대 금지 (이 절차 이전 단계)

- 파일 Write / Edit
- 외부 리서치 (WebSearch, WebFetch, 외부 MCP fetch)
- "계획" 또는 "방향성" 을 사용자에게 확정적으로 제시
- 텍스트로 질문을 나열한 뒤 응답을 기다리는 패턴 — 질문은 `AskUserQuestion` 도구로만

## Auto 모드·자동 실행 환경 처리

Auto 모드·자동 실행 정책이 활성화된 경우에도 이 규칙은 우선한다. 단, `intent-clarifier` 스킬의 에스컬레이션 절차에 따라 사용자가 2 회차에도 "알아서 해" 로 응답하면, 가장 단순·가역적 가정을 1줄로 명시 후 진행하는 것은 허용된다. 가정 명시 의무는 Auto 모드에서도 유지.

## 다른 항상적용 규칙과의 충돌

프로젝트에 "즉시 구현 시작", "질문 없이 진행", "빠르게" 류의 항상적용 규칙이 이미 있는 경우, **이 규칙이 우선한다.** 이유: 충돌하는 두 규칙 모두 `alwaysApply: true` 이지만, 의도 확인 없는 암묵 결정은 되돌리기 비용이 가장 큰 실수 유형이며 다른 규칙들이 이 시점에 의존하기 때문이다. 충돌 규칙을 발견하면 사용자에게 조정을 제안할 수 있으나, 그 제안 자체도 이 규칙의 절차를 따른다 (즉, 첫 턴 질문 없이 규칙 수정을 밀어붙이지 않는다).

## 이 규칙이 존재하는 이유

요청의 첫 턴에 잘못된 방향을 암묵적으로 고정하면, 이후의 리서치·계획·구현이 모두 그 방향 위에 쌓여 되돌리기 비용이 급격히 커진다. 분기와 맥락을 선행 확인하여 되돌림 비용을 "첫 턴의 질문 1회" 수준으로 묶는다.
